El cambio clave de v2.0

Una sola mejora elimina el mayor punto de error del proceso

🔑

CORREOS_BIBLIOTECAS — el corazón del cambio

En v1, el solicitante debía escribir manualmente el correo de la biblioteca prestadora — fuente de errores, correos incorrectos y solicitudes perdidas. En v2, ese campo desapareció del formulario. La función getCorreoBiblioteca() consulta una tabla interna con los 25 correos de la red y los resuelve automáticamente por nombre exacto de institución. Un solo punto de verdad para toda la red.

❌ v1 — El solicitante escribía el correo

  • ⚠️ Campo de texto libre para el correo de la biblioteca
  • ⚠️ Errores tipográficos frecuentes
  • ⚠️ Solicitudes perdidas por correo incorrecto
  • ⚠️ Sin validación del destinatario
  • ⚠️ Cada cambio de correo requería actualizar el form

✅ v2 — El sistema resuelve el correo solo

  • ✔️ El solicitante elige la institución de una lista
  • ✔️ getCorreoBiblioteca() resuelve el correo internamente
  • ✔️ Cero posibilidad de correo incorrecto
  • ✔️ Si no hay correo → estado Pendiente + alerta admin
  • ✔️ Actualizar correos no requiere tocar el formulario

CORREOS_BIBLIOTECAS — tabla interna

25 instituciones de la Red Reunir (Risaralda + Eje Cafetero)

⚙️ Cómo funciona

Un objeto JS con las 25 instituciones de la red. El solicitante elige el nombre de una lista desplegable; getCorreoBiblioteca() hace el match por nombre exacto y retorna el correo.

"Universidad Tecnológica de Pereira": "biblioteca@utp.edu.co", "Universidad Católica de Pereira": "biblioteca@ucp.edu.co", "SENA Regional Risaralda": "biblioteca.risaralda@sena.edu.co", // ... 22 instituciones más "Institución sin correo aún": null

🚨 Caso especial — sin correo

Si la institución tiene null como valor, el sistema no falla: la solicitud queda en estado Pendiente y se envía alerta automática con todos los datos a alabrada3@areandina.edu.co para gestión manual.

📌 Esto evita solicitudes perdidas para bibliotecas nuevas en la red o con correo desactualizado. Basta actualizar el objeto JS para activar el flujo automático.

Acciones por link — doGet(e)

Cada botón de correo es una URL al Web App. Sin login, sin contraseñas.

?action=aprobar

✅ Aprobar solicitud

Cambia el estado a Aprobada, notifica al solicitante y registra en log.

?action=rechazar

❌ Iniciar rechazo

Muestra paginaRechazo() con textarea para el motivo. Es paso 1 de 2.

?action=confirmar_rechazo

❌ Confirmar rechazo

Guarda el motivo, cambia estado a Rechazada y notifica al solicitante. Paso 2 de 2.

?action=confirmar_prestamo

📖 Confirmar recepción

Calcula fecha de devolución +15 días y crea evento en Google Calendar.

?action=confirmar_devolucion

📗 Registrar devolución

Borra el evento de Calendar y cierra el ciclo completo de la solicitud.

Funciones de mantenimiento

Herramientas para configurar y reparar el sistema sin tocar datos

setupInicial()

🔧 Configuración inicial

Crea las hojas Solicitudes y Bibliotecas, instala el trigger diario de vencimientos.

crearFormularioReunir()

📋 Crear formulario

Genera el Google Form desde cero con todos los campos y la lista de 25 instituciones.

ajustarFormularioReunir()

🔄 Actualizar formulario

Actualiza la lista de instituciones si cambian, sin recrear el formulario completo.

repararTriggerFormulario()

⚡ Reparar trigger

Borra triggers viejos y reconecta onFormSubmit si el flujo automático se interrumpe.

appendLog()

📜 Historial cronológico

Acumula en la columna S cada acción con fecha, actor y tipo de evento. Nunca sobreescribe.

verificarVencimientos()

⏰ Trigger diario — 8am

Recorre todos los préstamos activos: envía recordatorio a 3 días, marca vencida si pasó la fecha.
